# 2026-07-07 — quiz-answer-position-bias

**Severity:** medium
**Area:** pipeline (+ web/PDF surfaces via shared payload)
**Status:** fixed
**Keywords:** quiz, questions, correct_answer, options, shuffle, detail_enrich, positional-bias, all-B, MCQ

## Symptom

The admin recalled quizzes where "the choices are all B" — every question's
correct answer landed on the same option letter. Confirmed empirically on
2026-07-07 live content: across all 18 articles of the day (108 questions,
both `easy` and `middle` levels, fetched from
`redesign-daily-content/2026-07-07/article_payloads/payload_<sid>/<level>.json`),
the correct-answer position was **A 6% / B 71% / C 22% / D 0%**. A kid can
score 100% by always tapping the second option; option D is never correct.

## Root cause

Quizzes are generated by DeepSeek in `detail_enrich`
(`pipeline/news_rss_core.py`), which returns each question as
`{question, options:[4], correct_answer}` where `correct_answer` is a string
that must equal one option verbatim (prompt spec at
`pipeline/news_rss_core.py:1033`). The LLM has a strong positional bias — it
parks the correct answer in the same slot — and **nothing downstream ever
reorders `options`.** The frontend
(`website/article.jsx:117-119`) faithfully renders options in the emitted
order and derives the correct index by string match, so the model's bias
passes straight through to the rendered quiz (and to the printable PDF, which
reads the same `questions`). Nothing caught it because there was no test or
monitor on the answer-position distribution, and each individual quiz is
internally valid (`correct_answer` does match an option) — only the
*aggregate* distribution reveals the bias.

## Fix

PR: fix/quiz-answer-shuffle. Reorder each question's `options` at the single
choke point that feeds both web and PDF.

- `pipeline/quiz_shuffle.py` (new) — `shuffle_quiz_options(details)` permutes
  each question's `options` list in place with a deterministic per-(slot,
  question) seed. `correct_answer` is stored separately as a string, so
  reordering keeps the payload valid; the frontend re-derives the index by
  match. Questions whose `correct_answer` is not among options, or with < 2
  options, are left untouched (malformed data, logged — not silently
  reordered).
- `pipeline/news_rss_core.py:1547` — call `shuffle_quiz_options(details)`
  right after `filter_keywords`, inside `detail_enrich`, before the return.
- `pipeline/news_rss_core.py:32` — import `shuffle_quiz_options`.
- `pipeline/test_quiz_shuffle.py` (new) — pinning test.

Verified on real data: after the shuffle the 2026-07-07 distribution moves to
**A 31% / B 17% / C 19% / D 33%** (from 6/71/22/0).

Backend-only change — takes effect at the next daily pipeline run. Existing
already-published payloads keep their biased order until re-generated (see
Related for the optional backfill).

## Invariant

MCQ option order MUST be decoupled from the LLM's emission order before a
quiz is published. Any future change to `detail_enrich` must keep
`shuffle_quiz_options` (or an equivalent randomizer) between question
generation and payload write. `correct_answer` MUST remain a verbatim member
of `options` after any reordering, since both the web app and the PDF resolve
the correct choice by string match.

## Pinning test

`python -m pipeline.test_quiz_shuffle` (also runs under pytest). It fails
loudly if options stop being reordered (`test_breaks_positional_bias`), if a
shuffle drops/alters an option or `correct_answer`
(`test_preserves_option_set_and_correct_answer`), or if reordering stops
being reproducible (`test_deterministic_across_runs`).
